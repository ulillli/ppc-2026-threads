## 1. Контекст

Выбор технологии TBB (oneAPI Threading Building Blocks) обусловлен
необходимостью использования высокоуровневого планировщика задач. В отличие от
OMP, TBB позволяет реализовать более эффективное разбиение задачи на блоки и
эффективно распределять их между потоками, что критично для алгоритмов с
интенсивным доступом к памяти.

## 2. Постановка задачи

Задача — отсортировать массив из $10^7$ вещественных чисел типа `double`. В
качестве базовой линии (baseline) используется результат последовательной версии
из `seq/report.md` (время выполнения: 2058 мс).

Детали постановки задачи можно найти в `seq/report.md` в разделе
`2. Постановка задачи`.

## 3. Базовый алгоритм

Последовательная версия выполняет 8 проходов по всему массиву (по одному на
каждый байт 64-битного представления числа), используя промежуточную конвертацию
в `uint64_t` для корректного учета знакового разряда.

## 4. Схема распараллеливания

В данной реализации используется **блочный параллельный алгоритм**:

- **Примитив:** `tbb::parallel_for`.
- **Диапазон:** массив делится на блоки фиксированного размера
  (`block_size = 2048`). Такое значение выбрано для соблюдения баланса между
  накладными расходами на создание задач и локальностью данных в кэш-памяти.
- **Механизм:**
    1. **Параллельный подсчет:** каждый блок обрабатывается независимо,
       гистограмма байтов записывается в локальный аккумулятор
       `local_counts[block_idx]`. Это исключает состояние гонки без
       использования `atomic` или `critical`.
    2. **Последовательное сканирование:** вычисляются глобальные смещения
       (`offsets`) для каждого блока и каждого байта.
    3. **Параллельная перестановка:** элементы распределяются по целевому
       массиву параллельно по блокам.
- **Partitioner:** используется стандартный планировщик TBB, который
  автоматически распределяет блоки по доступным ядрам.

## 5. Детали реализации

- **Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.
- **Локальные аккумуляторы:** для каждого блока выделяется
  `std::array<int, 256>`, что предотвращает конфликты доступа.
- **Работа с памятью:** алгоритм использует двойную буферизацию (переключение
  между `bytes` и `out`), что минимизирует лишние копирования. Векторы
  `local_counts` и `local_offsets` аллоцируются один раз перед началом проходов.

## 6. Проверка корректности

Результат работы TBB-версии сравнивался с `std::sort` на 7 функциональных тестах
(положительные, отрицательные, смешанные числа, высокая точность, малые
значения, идентичные данные). Ошибок в логике сортировки при многопоточном
исполнении не выявлено. А также производилось сравнение с работой SEQ версии.

Набор функциональных тестов:

1. **test1:** положительные числа с дробной частью.
2. **test2:** отрицательные числа (проверка корректности работы битовой маски
   для знака).
3. **test3:** смешанные знаки и наличие нуля (проверка правильного расположения
   нуля относительно положительных и отрицательных чисел).
4. **test4:** числа с большой точностью и разным количеством знаков после
   запятой.
5. **test5:** группа очень близких значений (проверка чувствительности
   сортировки к младшим разрядам/байтам).
6. **test6:** экстремально малые значения (близкие к машинному нулю).
7. **test7:** массив идентичных элементов.

## 9. Выводы

TBB показал лучшее ускорение (до 1.17x) по сравнению с OMP благодаря более
глубокой параллелизации стадий подсчета и перестановки. Снижение эффективности
при увеличении числа потоков до 8 связано с ростом накладных расходов на
синхронизацию префиксных сумм между большим количеством блоков и ограничением
пропускной способности памяти.

TBB удобен для реализации сложных схем распараллеливания, требующих блочного
разбиения. Однако overhead планировщика и последовательный этап вычисления
смещений между блоками ограничивают масштабируемость на данной задаче.
