## 1. Контекст

Гибридная версия (MPI + OMP) требуется для обеспечения масштабируемости
алгоритма в гетерогенных средах. MPI позволяет распределять вычислительную
нагрузку между узлами сети, а OMP — максимально эффективно задействовать
многоядерность каждого отдельного процессора внутри узла. Это необходимо для
обработки массивов данных, объем которых превышает ресурсы одного
вычислительного узла.

## 2. Постановка задачи

Задача — отсортировать массив из $10^7$ вещественных чисел типа `double`. В
качестве базовой линии (baseline) используется результат последовательной версии
из `seq/report.md` (время выполнения: 2058 мс).

Детали постановки задачи можно найти в `seq/report.md` в разделе
`2. Постановка задачи`.

## 3. Базовый алгоритм

Последовательная версия выполняет 8 проходов по всему массиву (по одному на
каждый байт 64-битного представления числа), используя промежуточную конвертацию
в `uint64_t` для корректного учета знакового разряда.

## 4. Межпроцессная схема

- **Роли:** Rank 0 (Master) выполняет функции координатора: считывает входные
  данные, рассчитывает размеры смещений и выполняет финальное слияние. Остальные
  Rank-и выполняют локальную сортировку своих частей.
- **Распределение данных:** используется `MPI_Scatterv` для передачи кусков
  массива разного размера (если общее число элементов не кратно числу
  процессов).
- **MPI-вызовы:**
  - `MPI_Bcast`: передача общего размера массива всем процессам.
  - `MPI_Scatterv`: распределение исходного массива.
  - `MPI_Send` / `MPI_Recv`: сбор отсортированных локальных частей на Rank 0.
  - `MPI_Bcast`: финальная рассылка отсортированного массива всем процессам
      для синхронизации вывода.

## 5. Внутрипроцессная схема

Внутри каждого процесса используется технология **OpenMP**:

- **Параллелизация:** с помощью директив `#pragma omp parallel for`
  распараллелены этапы подготовки данных (`InBytes`) и восстановления исходных
  значений (`FromBytes`).
- **Деление работы:** массив данных внутри процесса делится планировщиком OpenMP
  на равные итерации между доступными ядрами.
- **Объединение:** локальные результаты внутри процесса записываются напрямую в
  выделенный буфер `uint64_t`, что не требует дополнительной синхронизации
  (каждый поток работает со своим индексом).

## 6. Детали реализации

- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.
- **Синхронизация:** использование `MPI_Bcast` в конце `RunImpl` гарантирует,
  что все процессы завершат работу одновременно и будут иметь доступ к полному
  результату.
- **Узкие места:**
    1. Передача больших объемов данных.
    2. **SimpleMerge**: на Rank 0 реализовано простое последовательное слияние
       принятых массивов, что создает значительный последовательный участок и
       ограничивает ускорение.

## 7. Проверка корректности

Корректность проверялась сравнением с SEQ на стандартном наборе из 7 тестов
(включая отрицательные числа и дубликаты). Проверка на согласованность показала,
что после финального `MPI_Bcast` все процессы возвращают идентичный
отсортированный массив.

## 8. Экспериментальная среда

- **Конфигурация:** 4 MPI-процесса.
- **Потоки:** 4, 8, 12 потоков OMP на каждый процесс.
- **Build Type:** Release (MSVC 19.37).
- **Команды запуска:** `mpiexec -n 4 ./ppc_perf_tests.exe --gtest_filter=*all*`

## 10. Выводы

Наблюдается замедление относительно последовательной версии. Цена коммуникации
(Scatter/Send/Recv) и последовательного слияния на мастере на данном объеме
данных ($10^7$) превышает выигрыш от параллельных вычислений. Рост числа потоков
внутри процесса дает лишь незначительное сокращение времени.

Гибридная схема MPI + OpenMP оправдана в распределенных системах (кластерах) при
работе с экстремально большими объемами данных, не помещающимися в память одного
узла. Для задач, умещающихся в ОЗУ одной машины, данная схема неэффективна из-за
оверхеда на межпроцессное взаимодействие. Для улучшения показателей требуется
замена «простого слияния» на параллельное или каскадное.
