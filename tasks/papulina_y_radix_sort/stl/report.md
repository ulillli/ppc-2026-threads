## 1. Контекст

Реализация на базе стандартной библиотеки потоков необходима для исследования
производительности многопоточности без использования высокоуровневых абстракций
(таких как планировщики TBB или директивы OpenMP). Это позволяет оценить
реальные накладные расходы на создание/завершение потоков и понять эффективность
прямого управления ресурсами процессора.

## 2. Постановка задачи

Задача — отсортировать массив из $10^7$ вещественных чисел типа `double`. В
качестве базовой линии (baseline) используется результат последовательной версии
из `seq/report.md` (время выполнения: 2058 мс).

Детали постановки задачи можно найти в `seq/report.md` в разделе
`2. Постановка задачи`.

## 3. Базовый алгоритм

Последовательная версия выполняет 8 проходов по всему массиву (по одному на
каждый байт 64-битного представления числа), используя промежуточную конвертацию
в `uint64_t` для корректного учета знакового разряда.

## 4. Схема распараллеливания

Реализована схема с разделением данных на непересекающиеся диапазоны:

- **Деление данных:** dесь массив делится на $P$ равных сегментов (где $P$ —
  число потоков). Каждый поток получает свой диапазон индексов `[start, end)`.
- **Действия потоков:** внутри каждого из 8 проходов потоки параллельно
  выполняют две фазы:
    1. **Подсчет частот:** каждый поток заполняет свою локальную таблицу частот
       (гистограмму) для закрепленного за ним диапазона.
    2. **Перестановка:** после вычисления глобальных смещений (выполняется
       последовательно) потоки параллельно переносят элементы из исходного
       массива в целевой.
- **Локальные результаты:** хранятся в двумерном векторе
  `std::vector<std::vector<size_t>> local_hists`, где каждый внутренний вектор
  принадлежит строго одному потоку.
- **Join:** выполняется дважды в рамках каждого байтового прохода (после
  подсчета частот и после перестановки). Метод `join()` гарантирует, что все
  потоки завершили свою часть работы перед переходом к следующей стадии.
- **Синхронизация:** используется неявная синхронизация через барьер `join()`.
  Явные мьютексы или атомарные операции не требуются, так как потоки пишут в
  независимые области памяти.

## 5. Детали реализации

- **Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`.
- **Карта поток -> диапазон:** реализована через функцию `GetThreadRange`,
  которая вычисляет границы сегментов на основе общего размера массива и индекса
  потока.
- **Схема объединения:** глобальные позиции элементов вычисляются в методе
  `ComputeOffsets`. Метод суммирует значения из всех `local_hists`, формируя
  таблицу `thread_pos`, которая указывает каждому потоку точное место для записи
  каждого типа байта.

**Ключевой фрагмент кода:**

```cpp
for (unsigned int t = 0; t < num_threads; ++t) {
  workers.emplace_back([&, t]() {
    auto range = GetThreadRange(size, num_threads, t);
    const auto *byte_view = reinterpret_cast<const unsigned char *>(src);
    for (size_t i = range.first; i < range.second; ++i) {
      int bucket = byte_view[(i * 8) + byte_idx];
      dst[thread_pos[t][bucket]++] = src[i];
    }
  });
}
for (auto &worker : workers) worker.join();
```

## 6. Проверка корректности

Результат работы STL-версии сравнивался с `std::sort` на 7 функциональных тестах
(положительные, отрицательные, смешанные числа, высокая точность, малые
значения, идентичные данные). Ошибок в логике сортировки при многопоточном
исполнении не выявлено. А также производилось сравнение с работой SEQ версии.

Набор функциональных тестов:

1. **test1:** положительные числа с дробной частью.
2. **test2:** отрицательные числа (проверка корректности работы битовой маски
   для знака).
3. **test3:** смешанные знаки и наличие нуля (проверка правильного расположения
   нуля относительно положительных и отрицательных чисел).
4. **test4:** числа с большой точностью и разным количеством знаков после
   запятой.
5. **test5:** группа очень близких значений (проверка чувствительности
   сортировки к младшим разрядам/байтам).
6. **test6:** экстремально малые значения (близкие к машинному нулю).
7. **test7:** массив идентичных элементов.

## 9. Выводы

Несмотря на высокие накладные расходы на создание и уничтожение потоков (16 раз
за всю сортировку), STL-версия показала наилучшее время выполнения. Это связано
с отсутствием сложности планирования, присущей TBB.

Ручная реализация через `std::thread` показала, что для задач с интенсивным
использованием памяти прямой контроль над потоками и явное разделение данных
могут быть эффективнее высокоуровневых библиотек. Основной выигрыш получен за
счет минимизации «скрытых» проверок и очередей задач, однако такая реализация
требует более тщательного написания кода для предотвращения ошибок доступа к
памяти.
