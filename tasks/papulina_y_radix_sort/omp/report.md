## 1. Контекст

Из последовательной версии (SEQ) переносится ядро алгоритма: побайтовая
сортировка (`SortByByte`) и битовые преобразования (`InBytes`, `FromBytes`).
Перенос в среду OpenMP необходим для разделения исходного массива на независимые
части и их параллельной обработки на многоядерной архитектуре.

## 2. Постановка задачи

Задача — отсортировать массив из $10^7$ вещественных чисел типа `double`. В
качестве базовой линии (baseline) используется результат последовательной версии
из `seq/report.md` (время выполнения: 2058 мс).

Детали постановки задачи можно найти в `seq/report.md` в разделе
`2. Постановка задачи`.

## 3. Базовый алгоритм

Последовательная версия выполняет 8 проходов по всему массиву (по одному на
каждый байт 64-битного представления числа), используя промежуточную конвертацию
в `uint64_t` для корректного учета знакового разряда.

## 4. Схема распараллеливания

В данной реализации используется следующая схема распараллеливания:

- **Декомпозиция данных:** вектор делится на $P$ равных частей (чанков) с
  помощью `std::span`.
- **Параллельная сортировка:** используется директива
  `#pragma omp parallel for`. Каждый поток получает свой чанк и выполняет в нем
  полноценную поразрядную сортировку (все 8 проходов).
- **Параллельное слияние:** после локальной сортировки части объединяются, на
  каждой итерации соседние пары чанков сливаются параллельно до тех пор, пока не
  останется один отсортированный массив.
- **Переменные:**
  - `shared`: исходный массив данных, векторы смещений и чанков.
  - `private`: индексы циклов (автоматически при `parallel for`).
- **Синхронизация:** явные барьеры в конце каждой итерации цикла слияния
  обеспечиваются завершением директивы `parallel for`. Атомарные операции не
  требуются, так как потоки работают с непересекающимися областями памяти.

## 5. Детали реализации

- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.
- **Изменения относительно SEQ:**
    1. Введен метод `MergeChunks`, реализующий параллельное слияние пар
       векторов.
    2. `RunImpl` теперь отвечает за логику разбиения массива на `std::span` и
       запуск параллельных секций.
- **Риски гонок:** устранены за счет жесткого закрепления индексов чанков за
  потоками. Каждый поток в `MergeChunks` пишет результат слияния в свой
  уникальный индекс нового вектора `next`.

## 6. Проверка корректности

Результат работы OMP-версии сравнивался с `std::sort` на 7 функциональных тестах
(положительные, отрицательные, смешанные числа, высокая точность, малые
значения, идентичные данные). Ошибок в логике сортировки при многопоточном
исполнении не выявлено. А также производилось сравнение с работой SEQ версии.

Набор функциональных тестов:

1. **test1:** положительные числа с дробной частью.
2. **test2:** отрицательные числа (проверка корректности работы битовой маски
   для знака).
3. **test3:** смешанные знаки и наличие нуля (проверка правильного расположения
   нуля относительно положительных и отрицательных чисел).
4. **test4:** числа с большой точностью и разным количеством знаков после
   запятой.
5. **test5:** группа очень близких значений (проверка чувствительности
   сортировки к младшим разрядам/байтам).
6. **test6:** экстремально малые значения (близкие к машинному нулю).
7. **test7:** массив идентичных элементов.

## 9. Выводы

OMP-версия дает выигрыш при обработке независимых чанков данных. Однако на
данной задаче выигрыш ограничен пропускной способностью памяти, так как Radix
Sort интенсивно обращается к ОЗУ. Основное узкое место — необходимость аллокации
памяти под копии чанков во время слияния, что нивелирует выигрыш от параллельных
вычислений на данном объеме данных.
